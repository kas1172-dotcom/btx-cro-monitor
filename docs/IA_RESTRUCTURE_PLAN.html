Product & architecture plan — for review, not yet built

Restructuring the cockpit into eight tabs a CRO can actually live in

Today the product runs on two overlapping navigation models (a flat surface list and a legacy "brain area" rail) and a single-user, 40-item local cache standing in for memory. This plan replaces both with one IA — Home, Clients, Prospecting, Trip Planner, Analysis, Deliverable Editor, HubSpot Viewer, Settings — plus a real backend for saved deliverables, one reusable "generate deliverable" wizard instead of three bespoke ones, and a ChatPill that finally knows what tab you're looking at.

00Architecture decisions

Every tab below leans on a small number of shared pieces. Building those once, first, is what keeps eight tabs from turning into eight slightly-different implementations of the same modal. These are the calls the rest of the plan assumes — flagged up front rather than buried, since a couple of them are genuine judgment calls rather than facts pulled from the code.

D1 — One navigation model, not two

The app currently runs SurfaceId (frontend/src/app/surfaces.ts) and the older BrainArea rail (frontend/src/app/railViews.ts) side by side, translated back and forth in store.ts. The eight-tab IA replaces both with a single TabId enum. RailAreaView, BrainHome, and the brainAreaForSurface/surfaceFromBrainArea translation layer get deleted, not deprecated.

D2 — Program memory becomes a real backend table

Right now "saved deliverables" is localStorage, one browser, capped at 40 documents (frontend/src/memory/localMemory.ts). A Deliverable Editor tab that browses a library, a Clients tab that saves a deck, and a Trip Planner that compiles a multi-stop brief all need the same durable store. This is the one piece of new backend infrastructure the whole plan depends on — a Postgres deliverables table via the existing Alembic/SQLAlchemy setup, keyed to canonical_account_id and an optional program_id/trip_id.

Why now: every one of tabs 2–6 either writes to or reads from this store. Building it per-tab would mean four incompatible half-versions of "memory."

D3 — Clients vs. Prospecting splits on business_motion, not relationship

The obvious field, Company.relationship (self|customer|supplier|competitor|target), is what Account360 already filters on today — but it can't tell "an existing customer we're expanding" from "a prospect we're actively closing." business_motion (manage_current_business|grow_existing_business|prospect_new_business|reduce_risk) already exists on every account, is already populated by the HubSpot mapper, and maps directly onto the CLAUDE.md current-business/prospecting split. Clients tab = manage_current_business | grow_existing_business | reduce_risk. Prospecting tab = prospect_new_business. A deal in active pipeline with an existing customer is a Clients-tab opportunity, not a Prospecting-tab one — that's the judgment call worth confirming.

D4 — One Deliverable Wizard, reused three times

Tabs 2, 3, and 5 each describe the same shape: pick deliverable type(s) → confirm context → generate → save / download / push-to-CRM → done. Building three bespoke modals would triple the surface area for a UI bug. This plan specs one <DeliverableWizard> component, parameterized by entry context (account, prospect, or "insert into an existing deliverable"), that replaces the ad-hoc createDeliverable() buttons in Dossier.tsx and the parameter popovers in AskBrainBar.tsx.

D5 — HubSpot gains write muscle it doesn't have today

btx_platform/hubspot.py can read companies/contacts/deals and do exactly one write: create_task(). A HubSpot Viewer that creates lists and a CSV importer that creates companies/contacts both need new batch-write methods. Scope and auth for these should be verified against current HubSpot API docs during implementation, not assumed from this plan.

D6 — "Request an integration" is a stored ticket, not a live pipe

Settings needs a way to say "add Outlook" without actually building an Outlook integration. Simplest honest version: a backend-stored request record (requester, integration name, notes, status) surfaced in a Settings list — no outbound email/Slack in v1. Open question for you: where should the request actually notify you? (See Risks.)

D7 — Rankings stay compact by default; explanation is one click away

Revised from the original CLAUDE.md rule that every ranking, signal, or score must show its full reasoning inline. That reads as clutter at list scale. New default: any scanning view (a prospect list, a trip's candidate list, a map pin) shows only a rank position, a short label, and a single compact confidence chip — no evidence paragraph, no always-on justification block. The full why-it-ranks-highly explanation, supporting evidence, and confidence detail sit behind one explicit drill-down (a "Why this ranks here" expand or detail-view open), never force-displayed before the user asks. This applies everywhere a ranked or scored list appears — Prospecting and Trip Planner are the two places below where the original acceptance criteria needed rewriting to match.

Why: the CRO scans lists to decide who to look at next; a wall of evidence per row slows that scan down without being read most of the time. The explanation still exists and is still one click away — it's the default visibility that changes, not whether reasoning is available at all.

Cross-workflow

01Home

Goal

One screen that answers "what changed since I last looked" in under ten seconds, with a way to go deeper immediately. Not a dashboard — a briefing.

Why this matters for the BTX CRO

Jamie doesn't open this product to browse; he opens it between calls. The home screen's only job is to make the first five seconds worth it — otherwise ChatPill and the tabs never get used.

Proposed experience

  • Greeting line — "Good morning, Jamie" (time-of-day aware, already trivial given existing date helpers in frontend/src/app/dateDefaults.ts).
  • Mini brief — three to five lines, not a grid. Reuses the existing "what changed" logic from TodayBrief.tsx but trims it to the top signals only, each with a one-line reason and a link into the tab where the detail lives (Clients, Prospecting, or Trip Planner).
  • Attention strip — a compact count row: N accounts need attention · N deliverables awaiting approval · N deadlines this week, each number a link, not a card.
  • ChatPill bar — anchored, always visible, pre-seeded with today's context so the first question doesn't need re-explaining what's already on screen.
  • Deliberately excluded: no KPI tiles, no chart. Those live in Analysis. Home stays a briefing, not a dashboard — the two shouldn't compete for the same five seconds.

Files affected

frontend/src/ui/surfaces/TodayBrief.tsx (trim/rebuild), frontend/src/app/workItems.ts (reuse useWorkItems views), new frontend/src/app/greeting.ts, store's activeHome field folds into the new TabId model (D1).

Acceptance criteria

  • Renders in under three visual "chunks" — greeting, brief, attention strip — plus the persistent ChatPill bar.
  • Every line item links to the tab/detail view that explains it; nothing on Home is a dead end.
  • No client-facing numbers invented — every line traces to an existing signal or work item.

Goal

Everything about managing existing revenue — health, risk, deadlines, and the ability to generate something to send — in one place, filtered to accounts where BTX already has a relationship.

Why this matters

This is the "keep the business we have" half of the CRO's job. It replaces the undifferentiated Account360 list (which today mixes customers and prospects) with a view that's unambiguously about current accounts.

Proposed experience

  • Industry updates panel — signals filtered to entities resolved (via canonicalAccounts.ts) to accounts with business_motion in the current-business set. Reuses ProgramContractTracker.tsx's filtering logic, scoped down.
  • Upcoming deadlines — contract renewals, task due dates, anything with a date field on a current account, sorted ascending. New, small — no deadline tracker exists today.
  • Account list + detail — the existing Account360 master/detail pattern, kept, but filtered by D3's rule.
  • Generate Deliverable button — opens the shared Deliverable Wizard (D4), scoped to the currently selected account: choose type(s) → confirm → generate → save to memory / download / push to CRM → Done closes the wizard.

New business

03Prospecting

Goal

Surface new-business signals, let the CRO drill into a single prospect without drowning in fields, and turn interest into a saved deliverable or a HubSpot task in as few clicks as possible.

Why this matters

This is the "find new business" half. It's the mirror of Clients, scoped to business_motion: prospect_new_business, and it's where the map-adjacent work (aerospace/defense signals, regional clustering) belongs per the existing CLAUDE.md map principles.

Proposed experience

  • Industry updates panel — signals resolved to prospect accounts or unlinked market/program-scope entities (via resolveSignalRelationships), same visual language as Clients' panel for consistency.
  • Prospect list — compact rows only: rank position, company name, one short status line, and a small confidence chip (a colored dot/pill, not a paragraph). No evidence text at this level — the list is for scanning, not reading.
  • Prospect list → detail view — select a prospect, open a detail view that is deliberately not the full Account360 depth. Default pane shows who they are, what changed recently, and a recommended next move — compact, no evidence dump. A single "Why this ranks here" expand (collapsed by default) reveals the full evidence and confidence reasoning behind D7's progressive-disclosure rule. Full contact list and deal history sit behind their own separate "see more," distinct from the why-ranked expand.
  • Action button — opens a picker: "Create deliverable(s)" and/or "Create HubSpot task" and/or "Add as new contact/prospect," multi-select, single confirm.
  • Post-generation state — once something's been created for a prospect, the button's label changes from "Generate" to "Navigate to Deliverables tab" for that prospect, so the CRO always has a clear next step instead of re-triggering generation by accident.

Further thinking — worth deciding now

What happens to a prospect once it converts? When a HubSpot deal closes or a company's business_motion flips to manage_current_business, it should quietly disappear from Prospecting and appear in Clients on the next data refresh — no manual "promote" action. Worth confirming this is desired rather than an explicit conversion button.

How many prospects show at once? Recommend capping the default list to the top 20 by rank with a "show more," rather than paginating everything — matches the existing surfaces' simple .map() pattern and avoids a premature virtualization build.

Files affected

New frontend/src/ui/surfaces/Prospecting.tsx, new frontend/src/ui/prospecting/ProspectDetail.tsx (the deliberately-shallow detail view), shared DeliverableWizard (multi-select variant), frontend/src/ui/surfaces/WorkItemList.tsx's HubSpot task flow (reused, not rebuilt).

Acceptance criteria

  • Prospect detail view fits without scrolling on a standard laptop viewport for the common case (no scroll wall of fields).
  • Prospect list rows show only rank, name, one status line, and a confidence chip — no evidence text visible until expanded (D7).
  • Every prospect's detail view has a "Why this ranks here" expand that reveals full evidence and confidence on demand — never gone, never forced.
  • The "Generate" → "Navigate to Deliverables" button-state transition persists across a page refresh (i.e., backed by D2's memory, not component state).

New business

04Trip Planner

Goal

Turn "I'm going to be in the Dallas–Fort Worth area next month" into a concrete, ranked itinerary of who to see and why, without opening the full map first.

Why this matters

A trip is a forcing function — a small number of high-value in-person conversations, geographically and time-boxed. Today's itineraryAgent.ts and the "Plan a trip" AskBrainBar action exist but produce a document, not a workflow with a map and per-stop deliverables. This closes that gap.

Defining "goals"

Rather than invent a new taxonomy, trip goals reuse the account business_motion field directly — so the same language describes an account, a signal, and a trip:

Grow existing accounts

Visit or deepen relationships with current customers in the region. Maps to grow_existing_business.

Manage current business

Check in on active programs, address renewal or delivery risk. Maps to manage_current_business / reduce_risk.

Prospect new business

Meet aerospace/defense/manufacturing prospects newly surfaced by signals in the region. Maps to prospect_new_business.

Event anchor (optional)

A conference or airshow name + date, narrowing the date window and adding "who else is likely there" context to the itinerary prompt.

Proposed experience

  1. Input template (first view, no map) — region or radius-from-address, date window, one or more goals (checkboxes above), optional event anchor, and a meeting-capacity constraint ("up to 6 meetings over 3 days").
  2. Generate — itineraryAgent (extended) produces a day-by-day skeleton plus a ranked candidate list matching goals + region + canonical accounts + live signals. Candidates carry the same compact rank + confidence-chip treatment as Prospecting (D7) — full why-ranked evidence stays behind the per-stop drill-in, not shown on the list or the map itself.
  3. Map view — after generation, ProspectMap renders pinned to only the identified candidates, not the full universe — a filtered view, reusing mapModel.ts. Pins show a confidence-tier color/size only; no label clutter on the map itself.
  4. Per-stop drill-in — click a pin or list card, get the same shallow detail view as Prospecting/Clients (reused, not rebuilt) with its collapsed "Why this ranks here" expand, and the same Deliverable Wizard to generate a meeting brief for that stop.
  5. Trip Brief — a compiled deliverable: the itinerary, each stop's meeting brief, key talking points, and logistics — built by composing per-stop meetingBriefAgent output into one document via the existing deck/document composition path, saved to program memory as a single artifact.

Files affected

New frontend/src/ui/surfaces/TripPlanner.tsx, new frontend/src/ui/trips/TripInputForm.tsx, extended frontend/src/agents/itineraryAgent.ts (accept goals/region/date-window/event-anchor inputs; return ranked candidates, not just prose), reused ProspectMap.tsx + mapModel.ts (add a candidate-set filter prop), reused ProspectDetail.tsx (from WP5) and DeliverableWizard (from WP2), new composition step in frontend/src/agents/meetingBriefAgent.ts or a new tripBriefAgent.ts for the compiled document.

Acceptance criteria

  • Map never renders before the input template is submitted.
  • Map pins and list rows show rank/confidence-chip only; no evidence text visible until a candidate is opened (D7).
  • Every candidate's detail view has a "Why this ranks here" expand tying its ranking to the selected goal(s), collapsed by default.
  • The Trip Brief deliverable contains one section per visited stop plus a top-level itinerary/logistics section, and saves as a single item in program memory.

Cross-workflow

05Analysis

Goal

A figure-creation hub: pick a client (or "all"), pick a chart type, build it in a clean modal, then either save it standalone or drop it into an existing deliverable.

Why this matters

Analysis is the one tab that's genuinely cross-workflow — a chart can support a Clients-tab renewal conversation or a Prospecting-tab pitch equally. The existing groundwork is unusually solid here: ChartSpec/computeChart/AnalysisView already model exactly this (metric + viz + rows/cols/color/filters), and the Deliverable type already has a chart-spec block kind — meaning "save this chart into an existing deck" is mostly wiring, not new data modeling.

Proposed experience

  1. Client dropdown — scope the figure to one account or leave it at the portfolio level.
  2. Figure/chart type picker — the four existing viz types (heatmap, trend, ranked_bar, retention_grid) presented as visual thumbnails rather than a bare <select>, since "based on prototypes" implies picking from what's proven to render well.
  3. Analysis creator window — a clean modal (not full-page, unlike today's AnalysisView) walking through data type (which MetricId) and presentation (rows/cols/color), live-previewing the result before commit.
  4. Save destination prompt — on Done, ask: save as a standalone saved view, or insert into an existing deliverable (dropdown of program-memory documents, D2). Confirm closes the window; the user can immediately start a new figure.

Further thinking

Two heatmap renderers currently coexist: AnalysisView's interactive Grid and deliverables/steelSignalTemplates.tsx's static RetentionEarningsHeatmap (built for document embedding via renderToStaticMarkup). The "save into a deliverable" flow is the natural point to consolidate these — the interactive spec should be able to render itself in both interactive and static-embed modes rather than maintaining two chart implementations.

Files affected

frontend/src/ui/analysis/AnalysisView.tsx (extract into a modal), new frontend/src/ui/analysis/FigureTypePicker.tsx, frontend/src/metrics/chartSpec.ts (add a static-render mode), frontend/src/deliverables/types.ts (wire the existing chart-spec block insertion path), frontend/src/deliverables/steelSignalTemplates.tsx (consolidate RetentionEarningsHeatmap into the shared spec renderer).

Acceptance criteria

  • Chart type selection shows a live thumbnail/preview per type, not a bare dropdown label.
  • "Insert into existing deliverable" round-trips: the chosen deliverable, reopened in the Deliverable Editor, shows the new chart in place.
  • Closing the save dialog returns the user to a fresh, empty analysis creator — never a dead end.

CSV → CRM upload

Goal

Let the user upload a client/sales list (CSV) and have it become real HubSpot companies/contacts, with a mapping step in between so nothing gets pushed blind.

Why this matters

Confirmed via search: there is no file upload, CSV parsing, or CRM-import capability anywhere in the codebase today — this is genuinely greenfield, not an extension of something existing. It's also the highest-risk feature here, since it writes external data into a live CRM.

Proposed experience

  1. Upload — file picker (CSV only, v1), client-side parsed with papaparse (small, standard, handles quoting/edge-cases correctly — worth the one new dependency rather than hand-rolling a CSV parser).
  2. Column mapping preview — show detected columns, let the user map each to a CRM field (company name, domain, contact name, email, etc.) before anything is sent anywhere. No silent auto-push.
  3. Dedupe check — run mapped rows through the existing canonicalAccounts.ts matching logic before creating anything, surfacing "this looks like an existing account" rather than creating duplicates.
  4. Confirm → push — batch-create via the new hubspot.py write methods (D5), with a per-row success/failure summary at the end.

Where does this live?

Recommend surfacing it inside Prospecting (as "Import list") rather than as its own tab — it's fundamentally a prospecting-list-building action, and a ninth top-level tab for a single-purpose flow would work against the "keep it minimal" goal. Worth confirming that placement.

Files affected

frontend/package.json (add papaparse), new frontend/src/ui/prospecting/ImportListModal.tsx, new backend route + hubspot.py batch-write methods (shared with D5/WP9), canonicalAccounts.ts (reused for dedupe, not modified).

Acceptance criteria

  • Nothing is pushed to HubSpot before the user explicitly confirms the mapped preview.
  • Rows that dedupe-match an existing canonical account are flagged, not silently duplicated.
  • A failed row doesn't block the rest of the batch; the summary shows exactly what succeeded and what didn't, and why.
